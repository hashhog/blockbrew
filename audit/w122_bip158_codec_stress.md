# W122 — BIP-158 GCS Codec Stress-Vector Audit (blockbrew)

**Wave**: W122 (DISCOVERY, not fix)
**Date**: 2026-05-17
**Impl**: blockbrew (Go)
**Verdict**: **BUG FOUND** — 2 P0-CDIV bugs in the GCS bit-stream codec
**Reference**: `bitcoin-core/src/streams.h::BitStreamWriter`,
`bitcoin-core/src/blockfilter.cpp`, BIP-158, haskoin commit `4a2de0f`
(W121 BUG-16 / W122 motivating bug).

## Summary

The blockbrew BIP-158 codec at `internal/storage/blockfilterindex.go` has **two**
mutually compounding bit-level defects that make every emitted filter
byte-incompatible with Bitcoin Core. Both are P0-CDIV: a peer fetching
`getcfilters` / `getcfheaders` / `getcfcheckpt` from a blockbrew node and
validating against Core-format filter headers will see a divergent
filter-header chain on every block past genesis.

The defects are masked by two existing-test patterns:

1. **Round-trip-only matching** (`TestGCSEncodeDecode`, `TestBIP158VectorsFilterMatch`).
   Encoder and decoder share the same buggy bit-order, so any filter
   blockbrew produces, blockbrew can read back. Matches a script against itself.
2. **Documented opt-out from byte-exact comparison**
   (`TestBIP158Vectors`, comment at `blockfilterindex_test.go:541-554`).
   The existing test deliberately does not compare filter bytes against
   `blockfilters.json`'s `BasicFilter` field, citing "vectors generated by an
   older Core build". The comment is **factually wrong** — see "Audit checklist
   item 4: regression on Core's blockfilters.json" below.

Stress tests added by this audit (`internal/storage/w122_gcs_stress_test.go`):

## BUG-1 — `bitStreamWriter.writeBits` truncates uint64 v on 64-bit shift overflow

**Severity**: P0-CDIV (consensus-incompatible behavior; affects on-disk + wire format)
**ID**: BLOCKBREW-W122-1
**Location**: `internal/storage/blockfilterindex.go:565-574`

```go
func (w *bitStreamWriter) writeBits(v uint64, n uint) {
    w.accumBits |= v << w.numBits   // (1)
    w.numBits += n
    for w.numBits >= 8 {
        w.bytes = append(w.bytes, byte(w.accumBits))
        w.accumBits >>= 8
        w.numBits -= 8
    }
}
```

When line (1) shifts `v` left by `numBits`, the **top `numBits` bits of `v`
fall off the high end of the uint64**. For the only large-`n` call site in
the codebase — `golombRiceEncode`'s unary-run loop calling
`writeBits((1<<count)-1, count)` with `count == 64` (line 595) — this means:

* If `numBits == 0` when the call fires → no truncation (v << 0 == v). OK.
* If `numBits == k > 0` → the top `k` bits of `v = 0xFF..FF` are dropped, so
  only `64 - k` ones make it into the bit stream. The encoded unary run is
  `64 - k` ones long instead of 64.

The decoder reads the same buggy stream and sees quotient `64 - k`, so the
recovered Golomb-Rice value is wrong by `k * 2^P = k * 524288` (for P=19).

**Reproducer** (`TestW122_UnaryRunsAfterPriorBits`):
```
Element 1: value = (1 << 19) | 5    // q=1, r=5   → 21 bits → leaves numBits=5
Element 2: delta = 64 << 19         // q=64, r=0  → wants 64 ones + 0 + 19 zero bits

Expected after decode of element 2: 64 << 19 = 33554432
Actual after decode of element 2:   59 << 19 = 30932992   (5 ones lost)
```

**Practical-impact analysis**: With M=784931 and P=19, the per-element delta
distribution is approximately geometric with mean ≈ M. The probability of any
single quotient being ≥ 64 is on the order of `2^-65` — vanishingly small for
random scripts but **achievable with adversarial element placement**. The bug
is latent today (never triggered by real testnet4 / mainnet blocks) but
**reproducible via a chosen-input attack** against a blockbrew node serving
filters: pick scripts whose siphash output places one delta in the q≥64 region
and the prior element's encoding leaves numBits > 0. The serving node then
emits a filter that no Core-compliant decoder can parse correctly.

**Cross-reference**: This is the **direct analog of haskoin BUG-16**
(commit `4a2de0f`) — the Word64 boundary issue that motivated W122. The
blockbrew shape differs in code structure (LSB-first accumulator vs haskoin's
MSB-first accumulator) but the root cause is identical: failure to bound-check
the implicit shift width.

## BUG-2 — Bit-ordering within byte is LSB-first; Bitcoin Core is MSB-first

**Severity**: P0-CDIV (every filter byte-incompatible with Core)
**ID**: BLOCKBREW-W122-2
**Location**:
* Encoder: `internal/storage/blockfilterindex.go:565-574` (`bitStreamWriter.writeBits`)
* Decoder: `internal/storage/blockfilterindex.go:731-745` (`bitStreamReader.readBits`)

### What Core does

`bitcoin-core/src/streams.h` lines 329-344, `BitStreamWriter::Write`:

```cpp
void Write(uint64_t data, int nbits) {
    while (nbits > 0) {
        int bits = std::min(8 - m_offset, nbits);
        m_buffer |= (data << (64 - nbits)) >> (64 - 8 + m_offset);
        m_offset += bits;
        nbits -= bits;
        if (m_offset == 8) { Flush(); }
    }
}
```

For input `Write(0b10101, 5)` starting from `m_offset = 0`:
* `m_buffer |= (0b10101 << 59) >> 56 = 0b10101000`

The first bit of the input lands in **bit 7** (MSB) of `m_buffer`. Subsequent
calls pack bits leftward → rightward within the byte, MSB-first.

### What blockbrew does

`internal/storage/blockfilterindex.go:565-574`:

```go
func (w *bitStreamWriter) writeBits(v uint64, n uint) {
    w.accumBits |= v << w.numBits
    w.numBits += n
    for w.numBits >= 8 {
        w.bytes = append(w.bytes, byte(w.accumBits))
        w.accumBits >>= 8
        w.numBits -= 8
    }
}
```

For input `writeBits(0b10101, 5)` starting from `numBits = 0`:
* `accumBits |= 0b10101 << 0 = 0b00010101`

The first bit of the input lands in **bit 0** (LSB) of `accumBits`. When
`byte(accumBits)` is emitted, that bit becomes the LSB of the output byte.
Blockbrew packs LSB-first within each byte.

### Direct demonstration

```
Sequence:    Write(0xA, 4), Write(0x5, 4)    // = 0b1010_0101
Core bytes:  a5    (MSB-first packing)
Blockbrew:   5a    (LSB-first packing)
```

```
Genesis vector (blockfilters.json height-0 entry):
    Core JSON BasicFilter:                  019dfca8
    blockbrew encodeGCS:                    0155fe0e   ← byte-incompatible
    coreEncodeGCS (reference, this audit):  019dfca8   ← matches Core
```

The reference encoder in `w122_gcs_stress_test.go` is the **same algorithm with
just the bit ordering corrected**, and it produces Core's exact bytes.

### Why round-trip tests miss it

Both the encoder and decoder are LSB-first, so they agree with each other.
Every test in `blockfilterindex_test.go` that encodes a filter, then asks
"do my known scripts match it?" passes — even though the bytes are wrong.

The only test that could have caught this — `TestBIP158Vectors` — is
deliberately neutered (lines 541-554):

> The blockfilters.json test vectors were committed to Bitcoin Core in 2018
> ... The specific filter-byte values in the JSON ... do NOT match what the
> current Bitcoin Core codebase produces ... Because of this discrepancy,
> this test does NOT compare filter bytes against the JSON.

This rationalization is **incorrect**. Core's own
`src/test/blockfilter_tests.cpp::blockfilters_json_test` (line 172) performs
exactly the byte-exact comparison the blockbrew test refuses to do:

```cpp
BlockFilter computed_filter_basic(BlockFilterType::BASIC, block, block_undo);
BOOST_CHECK(computed_filter_basic.GetFilter().GetEncoded() == filter_basic);
```

Every Bitcoin Core CI run validates `blockfilters.json` byte-for-byte. The
vectors are accurate. blockbrew's filter encoding is what differs.

### Cross-impl confirmation

`rustoshi/crates/storage/src/indexes/gcs.rs` (lines 418-455) uses MSB-first
packing (`self.current_byte |= 1 << (7 - self.bit_pos)`) — matching Core.
blockbrew is the outlier.

### Impact

Every BIP-158 filter served by a blockbrew node — via the BIP-157 `cfilter` /
`cfheaders` / `cfcheckpt` P2P messages, or via REST/RPC endpoints if any — is
byte-incompatible with what a Bitcoin Core peer or Neutrino-style light client
expects. Specifically:

* `getcfilters` response bytes are the wrong filter; the requesting peer's
  `f_hash = SHA256d(filter_bytes)` will not match the `f_hash` field of the
  `cfheaders` message claimed by the blockbrew node (which is computed over
  the same wrong bytes — so self-consistent), but any peer that **also** has
  the filter from a Core source will see a mismatch.
* `getcfheaders` chain headers therefore diverge from Core's published filter
  header chain after height 0. Cross-node BIP-157 service is broken.
* On-disk filter index serialization (`BlockFilterData`) is also LSB-first.
  Operators cannot migrate filter data between blockbrew and Core nodes
  without re-indexing from scratch.
* W121 BUG-7 (P0-CDIV `getcfcheckpt` off-by-one) compounds with this bug to
  produce *two* layers of wire incompatibility — but BUG-7 fixed in
  commit `4cee2c6` (FIX-74) is structural; this codec bug is per-byte.

## Recommended fix shape (informational — this is DISCOVERY)

Rewriting both `bitStreamWriter.writeBits` and `bitStreamReader.readBits` to
pack MSB-first, mirroring `streams.h`'s `BitStreamWriter`. The reference
encoder in `w122_gcs_stress_test.go::coreBitStreamWriter` (lines 49-83) is a
working byte-exact mirror — a fix can lift the same arithmetic.

While fixing, also address BUG-1: bound the per-iteration `count` correctly
(Core's loop never calls `Write` with `nbits == 64` AND `m_offset > 0`
simultaneously because the per-iteration `bits = min(8 - m_offset, nbits)`
clamp prevents it; blockbrew's choice to expose `writeBits(..., 64)` as a
single call needs the same internal byte-chunking).

A by-byte mirror of Core's algorithm fixes both bugs in one rewrite.

**Suggested follow-up wave**: FIX-NN-bip158-codec-msb-first. Single-file change
to `blockfilterindex.go`; delete the rationalizing comment block in
`blockfilterindex_test.go:541-554` and replace with byte-exact assertions
against `blockfilters.json`.
